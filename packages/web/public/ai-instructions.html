OWOX Model Canvas — OKF authoring guide

Instructions for an AI agent (Claude, Codex, Gemini, …) to generate a data model in OKF that imports cleanly into OWOX Model Canvas and pushes to OWOX Data Marts in one click.

Goal. Produce one or more Markdown documents in the format below. The user pastes or uploads them into Import OKF on the canvas, reviews the rendered graph, then clicks Push to OWOX to create draft Data Marts and joinable relationships.

1. What is OKF

Open Knowledge Format (OKF) is a vendor-neutral way to represent metadata as a directory of Markdown files with YAML frontmatter. Reference: github.com/GoogleCloudPlatform/knowledge-catalog (see the okf/ spec). This canvas uses a small authoring profile of OKF: one concept document per Data Mart, plus an owox: frontmatter block and a ## Joins section that the canvas understands.

2. Bundle structure

A model is a set of Markdown files — one per Data Mart. You may output them as separate files, or as a single pasted document where each file is preceded by a marker comment:

<!-- ecommerce/customers.md -->
---
...frontmatter...
---
# Customers
...

<!-- ecommerce/orders.md -->
---
...frontmatter...
---
# Orders
...

When pasting, the <!-- path/file.md --> markers separate documents. When uploading, just provide one .md file per Data Mart.

3. Anatomy of a Data Mart document

---
type: "OWOX Data Mart"
title: "Orders"
description: "One row per order"
tags: ["owox", "sql"]
timestamp: 2026-06-19T10:00:00Z
owox:
  key: orders                 # local id used to wire relationships (unique in the model)
  inputSource: SQL            # SQL | CONNECTOR | VIEW | TABLE  (default SQL)
  storageId: ""               # leave empty — the canvas sets the selected storage
  status: DRAFT
  id: null                    # leave null — filled after Push
  position: { x: 0, y: 0 }    # optional; the canvas auto-lays-out on import
---

# Orders

One row per order, used for revenue and retention.

## Schema

| Column | Type | PK |
|--------|------|----|
| `id` | STRING | ✓ |
| `customer_id` | STRING | |
| `order_date` | DATE | |
| `total` | FLOAT | |

## Definition

```sql
SELECT id, customer_id, order_date, total FROM `project.dataset.orders`
```

## Joins

- [Customers](./customers.md) — `customer_id = id`

Frontmatter fields

  • type — always "OWOX Data Mart".
  • title — human title (shown on the node).
  • description — one-line summary (optional).
  • owox.key — a short stable local id (e.g. orders); relationships reference it. Must be unique within the model.
  • owox.inputSource — one of SQLCONNECTORVIEWTABLE. Defaults to SQL.
  • owox.storageId — leave empty; the canvas applies the storage the user selects (one storage per model).
  • owox.position — optional; omit or { x: 0, y: 0 } — the canvas auto-arranges.

4. Output schema (fields)

Define the mart's columns in a ## Schema table: Column, Type, PK (put ✓ for primary keys). These are pushed to the Data Mart's output schema.

Allowed types (cross-storage canonical set — do not use others like DATETIME):

STRINGINTEGERFLOATNUMERICBOOLEANDATETIMETIMESTAMPBYTESGEOGRAPHYVARIANT

5. Definition (optional)

Optionally include a ## Definition fenced code block. Its meaning depends on inputSource:

  • SQL — a SQL query (use a ```sql fence), e.g. SELECT … FROM `project.dataset.table`.
  • TABLE — a fully-qualified table name, e.g. project.dataset.table.
  • VIEW — a fully-qualified view name, e.g. project.dataset.view (a reference, not a SQL query).
  • CONNECTOR — omit; configured in OWOX after creation.

6. Relationships (Joins)

Declare joinable relationships in a ## Joins section on the source mart. Each line links to the target document and states the join condition:

## Joins

- [Customers](./customers.md) — `customer_id = id`
- [Products](./products.md) — `product_id = id`
  • Format: - [Target Title](./target-slug.md) — `leftField = rightField`. Multiple keys: comma-separate the backticked pairs.
  • The filename slug is the target title lowercased with spaces → hyphens (e.g. Order Items → order-items.md).
  • Join fields should exist in both marts' ## Schema. If they don't, the canvas auto-adds them (as STRING) — but it's cleaner to define them.
  • Bidirectional relationship: add a matching Joins line in both documents (with the key sides swapped). The canvas renders it as a double-headed arrow.

8. Workflow

  1. Generate the documents above for the user's domain.
  2. User opens Import OKF, pastes the documents (with <!-- path --> markers) or uploads the .md files, clicks Import.
  3. The canvas renders nodes + relationships; the user reviews and tweaks.
  4. User picks a Storage and clicks Push to OWOX → draft Data Marts, output schemas and joinable relationships are created in the OWOX project.
Rules of thumb. One storage per model. Keep owox.key unique and stable. Use only the allowed schema types. Reference real fields in joins. Leave storageId/id empty — the canvas fills them.
